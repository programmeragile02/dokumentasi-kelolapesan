Source of truth

  • Central DB: tenant connection registry, instance mapping/runtime snapshot, API credentials, global queues, admin domain.
  • Tenant DB: instances, contacts, lists, chat, bots, sequences, blast, validator, attachment.
  • WA disk: LocalAuth session dan tracker ACK sementara—bukan business source of truth.

Trust boundaries

  • User route: JWT dalam cookie httpOnly.
  • Sevenlead ↔ WA: x-api-key.
  • Sevenlead ↔ socket: x-api-key; browser memakai JWT realtime.
  • External instance API: Bearer key per instance.
  • Membership provisioning: Bearer secret terpisah.

Alur database produk KelolaPesan

KelolaPesan berjalan sebagai produk multi-tenant: Membership menentukan lisensi dan paket, Sevenlead membuat registry koneksi di central DB, lalu semua data operasional user disimpan di database tenant milik workspace tersebut.

Central DB untuk apa?

Sebagai registry dan dispatcher lintas tenant: menyimpan daftar tenant, konfigurasi koneksi database tenant, mapping instance ke tenant, credential external API, mapping Instagram, snapshot runtime, onboarding, admin, dan pointer queue global.

Tenant DB untuk apa?

Sebagai source of truth operasional workspace KelolaPesan: instance WhatsApp, contact/list, chat, bot, sequence, blast, attachment, phone validator, Instagram/social account, automation, event, dan log pesan.

Koneksinya bagaimana?

Request user membawa session tenantId + licenseCode. Service mengambil config dari central, membuat tenant-specific Prisma client via getTenantPrisma(tenantId), lalu query hanya ke DB tenant sesuai scope.

Konsep yang wajib dipahami

Shared workspace

Satu tenant, satu DB

License agency tidak memanggil create_db lagi. Semua seat/license agency menunjuk tenantId dan tenant DB milik workspace yang sama.

Boundary root

Direct scope

Attachment, chat conversation, social account/automation, dan credential external API menyimpan atau menyalurkan agency scope untuk lookup dan realtime yang aman.

  1. User memasukkan username dan license code workspace.
  2. Membership memvalidasi produk/subscription dan mengembalikan mode=REGULAR, tenant, dbConfig, package, serta features.

B. Login AGENCY
  1. User memasukkan agency license code yang diberikan workspace owner/Membership.
  2. Membership memvalidasi seat/license dan mengembalikan mode=AGENCY, workspace tenantId, agency metadata, feature, quota, status, dan expiry.

D. Perubahan quota, status, label, atau expiry agency

Membership harus memanggil endpoint sinkronisasi setelah perubahan efektif:

POST /api/v1/external/membership/agency-license-sync
Authorization: Bearer <MEMBERSHIP_API_SECRET>
Content-Type: application/json

{
  "tenantId": "<WORKSPACE_TENANT_ID>",
  "licenseCode": "<AGENCY_LICENSE_CODE>"
}

1. Invalidate dan refresh

Cache tenant/license dihapus, lalu context terbaru diambil langsung dari Membership.

2. Reconcile quota

Jika jumlah instance melebihi limit, instance dalam scope agency dihapus permanen. Prioritas removal: DISCONNECTED → QR_READY → PENDING → AUTHENTICATED → CONNECTED; pada status yang sama, instance terbaru dihapus lebih dahulu.

3. Reconcile access

Jika status tidak ACTIVE atau expiry telah lewat, session WhatsApp instance connected dibersihkan.

4. Reconcile timeout

connectionExpiresAt menjadi tanggal paling awal antara batas durasi FREE/TRIAL dan access expiry dari Membership.

5. Return summary

Response memuat mode, status, expiry, display name, instance limit, jumlah instance deleted/adjusted/active.

Operasi destruktif: penurunan quota dapat memanggil permanent deletion, bukan sekadar disable. Membership/UI harus meminta konfirmasi yang sesuai sebelum mengurangi quota, dan operator harus memantau response reconciliation.
E. Worker, blast, sequence, dan subscription expiry
  • Blast dan sequence mengambil license dari instance.agencyLicenseCode; jika null, gunakan license workspace.
  • Access result dicache 60 detik per tenantId + licenseCode, tetapi worker dapat memaksa refresh.
  • Worker subscription mengelompokkan instance menjadi scope workspace (__workspace__) atau tiap agency license.
  • Jika satu scope tidak aktif, hanya recipient blast, sequence job, dan session instance pada scope tersebut yang dibatalkan/dibersihkan.
  • WA service tetap tidak mengenal agency. Ia menerima instanceId setelah Sevenlead menyelesaikan authorization.
F. Realtime dan pencegahan kebocoran event
  • Token realtime berlaku 15 menit dan membawa accessMode serta license context dari session.
  • REGULAR bergabung ke room workspace; AGENCY hanya ke room license miliknya.
  • Event resource agency dipublish ke room agency dan, bila includeWorkspace=true, juga ke room workspace owner.
  • Jangan menerima room name mentah dari client dan jangan broadcast event tenant tanpa agencyLicenseCode ketika event hanya boleh terlihat oleh satu agency.

Checklist menambah module baru

  1. Tentukan root ownership: instance, contact list, blast, sequence, atau resource lain.
  2. Semua user-facing service menerima { tenantId, licenseCode }, bukan tenantId saja.
  3. Gunakan TenantResourceScopeService; jangan membuat pengecekan ID yang hanya memvalidasi tenant.
  4. Pastikan list, detail, create, update, delete, import, export, retry, dan background job semuanya scope-aware.
  5. Jika resource tidak aman melalui relasi instance, simpan agencyLicenseCode secara eksplisit dan beri index.
  6. Propagasikan scope ke realtime event, cache key, external credential, attachment, dan audit log.
  7. Uji minimal: REGULAR melihat workspace; agency A melihat A; agency A tidak dapat membaca/mutasi ID milik agency B.
Contract risk yang harus diuji: SubscriptionAccessService dan reconciliation memakai top-level TenantContext.expiredAt. Membership harus mengisinya dengan effective expiry yang benar untuk agency—tanggal paling awal dari subscription workspace, expiry agency, dan expiry seat. Field metadata agency.effectiveExpiresAt tidak digunakan langsung oleh access checker saat ini.
06 · Social integration

Instagram dan Meta

Integrasi ini memakai Instagram API with Instagram Login untuk menghubungkan akun profesional ke satu instance Sevenlead, membaca media, menerima webhook komentar, lalu mengirim private reply. Bagian ini membedakan perilaku yang benar-benar ada di kode dari target production yang masih harus dibangun.

Implemented

OAuth + encrypted token

User menghubungkan akun Business/Creator melalui OAuth. Access token disimpan terenkripsi AES-256-GCM di tenant database.

Implemented

Comment-to-DM

Webhook komentar dicocokkan ke automation aktif. Match pertama mengirim private reply ke pemilik komentar dan menyimpan event serta delivery log.

Implemented

Text inbox

Event DM disimpan di durable inbox terenkripsi, diproses worker secara idempotent, lalu dipublish realtime. Media dicatat sebagai unsupported untuk rollout berikutnya.

  2. Meta Developer App dengan produk Instagram API dan Instagram Login yang dikonfigurasi.

Operational rule

Aturan deployment

  • Redirect/callback production harus HTTPS dan dapat dijangkau Meta.
  • Jangan menggunakan secret development di production.
  • Pin versi Graph API; lakukan regression test sebelum upgrade.
  • Tambahkan callback URI untuk setiap environment secara eksplisit.
  • Meta dashboard dan requirement review dapat berubah; validasi ulang terhadap dokumentasi resmi saat release.

Release, smoke test, dan troubleshooting

Release gate

  • Central dan tenant migration Instagram sudah terpasang pada semua tenant.

Diagnosis cepat

  • OAuth gagal: cek redirect URI, app ID/secret, state expiry, dan clock server.
  • Connected tetapi tidak ada event: buka diagnostics subscription, cek webhook fields, mode app, mapping IG user, dan signature logs.
  • Automation tidak match: cek media ID, status aktif, keyword normalization, dan urutan rule.
  • DM gagal: cek permission, token expiry, Graph API response/rate limit, serta reply log FAILED.
  • Agency tidak bisa akses: verifikasi agencyLicenseCode instance sama dengan license session.

Pattern queue yang dipakai

Central pointer queue

Central menyimpan tenantId + resourceId + scheduledAt + status. Worker tidak lagi scan seluruh database tenant. Detail dan monitoring tetap dibaca dari tenant DB.

Double claim

Job di-claim di central, lalu resource tenant di-claim lagi. Jika state tenant berubah, pointer central direkonsiliasi menjadi pending/paused/sent/failed/canceled.

Scaling rule: worker blast/sequence/validator aman untuk beberapa proses selama semuanya memakai central claim dengan FOR UPDATE SKIP LOCKED. Namun throughput outbound tetap dibatasi oleh ownership WA instance dan belum ada rate limiter global per nomor/instance.

Create/publish

Validasi instance connected, ownership agency, attachment, recipient source, batch allocation. DRAFT tidak membuat recipient. Publish menghasilkan status SCHEDULED dan recipient PENDING dalam transaction tenant.

Scheduling

Recipient dibagi ke explicit batch atau legacy batch size. Di dalam batch, jadwal berikutnya bertambah random antara minContactDelayMs dan maxContactDelayMs. Antar-message memakai delay tersendiri.

Claim

Central claim due pointer dengan row lock; tenant claim hanya menerima blast SCHEDULED/PROCESSING dan recipient PENDING due. Lock yang lebih lama dari 10 menit dianggap stale.

Execution

Worker cek subscription per workspace/agency, resolve watermark, render placeholder kontak, build attachment, lalu mengirim bubble berurutan. Recipient diproses sequential dalam satu worker batch.

Retry/finalize

Error retryable kembali PENDING pada +5 atau +30 menit. Attempt ke-3 terminal FAILED. Blast menjadi COMPLETED bila tidak ada pending, atau COMPLETED_WITH_ERRORS bila terdapat failed.

ACK/read

WA tracker mengasosiasikan message ID dengan recipient selama 7 hari. ACK ≥2 mengisi deliveredAt; event READ mengisi readAt. Realtime update dipublish setelah tenant state tersimpan.

Pause

Tenant recipient dan pointer central menjadi PAUSED; PROCESSING yang terlihat oleh executor dilepas.

Resume

Seluruh waktu pending/batch digeser relatif terhadap jadwal pending pertama agar interval tetap terjaga.

Stop/delete

Pending/processing dicancel; central pointers dan message logs diselaraskan. Data bisnis menggunakan soft delete.

Semantik delivery penting: worker menandai SENT sesudah HTTP send sukses, tetapi crash setelah WhatsApp menerima pesan dan sebelum status tersimpan dapat menyebabkan retry dan duplikasi. Saat ini belum ada end-to-end idempotency key yang dipaksakan oleh WA service.

11 · Data architecture

Database dan isolasi tenant

Produk KelolaPesan memakai pola central registry + database per tenant. Central membantu aplikasi menemukan tenant dan mengatur pekerjaan lintas database; tenant DB menyimpan data bisnis harian milik workspace. Dengan pola ini, data customer antarworkspace tetap terisolasi walaupun aplikasi dan worker-nya sama.

Central models

Tenant, UserOnboardingProgress, ExternalInstanceApiCredential, InstanceMapping, InstagramAccountMapping, InstanceRuntimeSnapshot, BlastRecipientJob, ContactListSequenceJobQueue, PhoneValidationJobQueue, policy settings, Admin, AdminWhatsappConnection, AdminNotificationJob, AdminBroadcast + recipient.

Tenant models

Instance, social/Instagram automation, ContactList/Field/Contact, ChatConversation/Message/Exclusion, MessageLog/Attempt, Bot/Scope/Action/Message, Sequence/Job, Blast/Batch/Message/Recipient, PhoneValidationJob/Item, Attachment.

Connection resolution

  1. User login menghasilkan session berisi tenantId, licenseCode, dan mode akses.
  2. Service mengambil tenant registry dari central DB.
  3. getTenantPrisma(tenantId) membangun koneksi ke database tenant yang benar.
  4. Service menjalankan query tenant dengan validasi ownership/scope.
  5. Jika perlu event realtime atau queue, pointer global disimpan di central setelah state tenant konsisten.

Data placement rule

  • Central: metadata global, lookup lintas tenant, credential, mapping, queue pointer, admin, dan status runtime.
  • Tenant: data operasional user/workspace, konten pesan, kontak, automation, log, dan resource yang perlu isolasi customer.
  • Jangan membaca tenant DB berdasarkan parameter bebas dari client; selalu resolve dari session dan central registry.
  • Jangan menyimpan payload bisnis besar di central jika cukup disimpan di tenant lalu central hanya menyimpan pointer.

Aturan isolasi

  • Jangan membuat Prisma tenant global tunggal; resolve dari central registry.
  • Cache key client memakai host + port + db + username; password tidak ditampilkan di log.
  • Worker melepaskan client tenant setelah selesai agar pool tidak terus bertambah.
  • Resource operasional aktif wajib memfilter deletedAt: null.
  • Mode AGENCY membatasi instance dengan agencyLicenseCode; resource turunan mengikuti ownership chain.
  • Cache context wajib per tenantId + licenseCode, bukan tenant saja.
Connection budget: default tenant connection limit adalah 2 per cached Prisma client. Total koneksi kasar = jumlah proses aplikasi/worker × tenant client aktif × connection limit. Pantau activeTenantClients dan kapasitas PostgreSQL sebelum menaikkan concurrency worker.
12 · Development migration

Migrasi schema saat development

Dua schema, dua migration history. Perubahan registry tenant, credential external API, mapping Instagram/WA, dan global queue masuk prisma/central. Perubahan data operasional workspace seperti instance, contact, blast, sequence, inbox, dan automation masuk prisma/tenant.
Setiap penambahan struktur wajib lewat script migration. Jika menambah tabel/kolom/enum/index di central, update prisma/central/schema.prisma lalu jalankan script central migration. Jika menambah struktur tenant, update prisma/tenant/schema.prisma, buat migration tenant, lalu siapkan rollout production dengan script tenant deploy. Jangan ubah struktur database manual tanpa migration yang tercatat di repo.

Central schema

Dipakai untuk satu database pusat. Migration cukup dijalankan sekali per environment.

npm run prisma:migrate:central -- --name <nama_migration>
npm run prisma:generate:central
npm run typecheck

Tenant schema

Dipakai untuk template semua database tenant. Migration dev dibuat sekali, lalu production rollout memakai script tenant deploy.

npm run prisma:migrate:tenant -- --name <nama_migration>
npm run prisma:generate:tenant
npm run typecheck

Checklist developer

  1. Tentukan target schema: central atau tenant. Jangan menaruh data operasional tenant di central kecuali memang berupa pointer/queue global.
  2. Pastikan strategi kompatibilitas: additive dulu untuk field baru, hindari langsung NOT NULL tanpa default/backfill.
  3. Edit schema secara minimal dan buat migration dengan nama deskriptif.
  4. Review file SQL di folder migration: potensi lock, table rewrite, drop column, rename, enum change, index besar, dan foreign key.
  5. Uji migration pada database lokal yang punya data representatif, bukan database kosong saja.
  6. Generate Prisma client sesuai schema yang berubah, lalu jalankan typecheck/lint/build sesuai scope release.
  7. Jika tenant schema berubah, tulis catatan release: fitur terdampak, tabel berubah, command production, batch size awal, dan smoke test.

Development smoke test

  • Login REGULAR dan AGENCY jika migration menyentuh resource tenant.
  • Jalankan minimal satu flow create/read/update pada tabel baru atau field baru.
  • Pastikan worker yang membaca schema baru tidak crash saat data lama masih kosong.
  • Untuk perubahan queue, uji claim/retry satu job sample.

Larangan normal flow

  • Jangan memakai prisma db push sebagai workflow development/release normal.
  • Jangan edit migration lama yang sudah pernah masuk branch/release bersama.
  • Jangan membuat migration destruktif tanpa rencana expand-contract dan backup.
  • Jangan deploy kode yang langsung mewajibkan semua tenant sudah berada di schema terbaru.
Catatan tenant. Endpoint membership create_db sudah memakai migrate deploy, sehingga tenant baru otomatis punya schema dan history migration yang benar. Untuk tenant existing di production, migration tetap harus di-rollout ke semua DB tenant melalui script tenant:schema:deploy.
13 · Production runbook

Migrasi production untuk seluruh tenant

Production memakai dua tahap: deploy migration central sekali, lalu rollout migration tenant ke semua database tenant. Karena tenant sudah up-to-date, workflow normal tidak memakai tenant:schema:sync; script itu hanya disimpan untuk legacy repair/catch-up bila suatu hari ditemukan tenant drift.

Rule utama production. Perubahan tenant schema tidak selesai hanya dengan deploy aplikasi. Setelah code release, jalankan tenant:schema:deploy sampai semua tenant DB menerima Prisma migration terbaru, lalu audit ulang hasilnya.

Jika struktur central berubah

Contoh: tambah tabel mapping global, credential, queue pointer, admin policy, atau registry field. Migration dijalankan sekali ke central DB sebelum/bersamaan dengan deploy app kompatibel.

npm run prisma:deploy:central
npm run prisma:generate:central

Jika struktur tenant berubah

Contoh: tambah field instance, contact, blast, inbox, automation, atau table operasional lain. Migration harus dijalankan ke semua tenant DB melalui script rollout.

npm run tenant:schema:deploy -- --limit=10 --offset=0 --apply --stop-on-error --json
npm run tenant:schema:audit -- --limit=10 --offset=0 --json

A. Preflight sebelum deploy

Review teknis

  • Pastikan migration SQL sudah direview: lock, rewrite table, enum, index besar, foreign key, drop/rename, dan NOT NULL.
  • Gunakan pola expand-contract untuk breaking change: tambah kolom/tabel dulu, deploy app kompatibel, backfill, verifikasi, baru contract di release berikutnya.
  • Pastikan code yang akan naik aman saat sebagian tenant masih schema lama selama rollout sequential.
  • Siapkan daftar smoke test fitur yang menyentuh tabel berubah.

Operational gate

  • Backup/PITR aktif dan restore drill tersedia.
  • Maintenance window atau low-traffic window disepakati untuk migration berisiko.
  • Owner release, owner database, dan incident commander jelas.
  • Monitoring app/worker/DB siap: error rate, latency, connection pool, lock, failed job, dan worker retry.
# Audit tenant sebelum rollout
npm run tenant:schema:audit -- --limit=25 --offset=0 --json

Output audit disimpan sebagai artifact release. Status ideal sebelum deploy normal adalah tenant tidak drift secara struktural; jika ada failed atau drift tidak terduga, investigasi dulu sebelum lanjut.

B. Urutan deploy normal

1. Deploy central migration

Jalankan sekali untuk database pusat karena central menyimpan tenant registry, mapping, credential, dan queue global.

npm run prisma:deploy:central

2. Deploy aplikasi kompatibel

Naikkan code yang toleran terhadap schema lama dan baru. Ini penting karena rollout tenant berjalan batch/sequential.

3. Canary tenant

Mulai dari tenant internal/sample. Pakai --stop-on-error agar proses berhenti saat satu tenant gagal.

npm run tenant:schema:deploy -- --tenant-ids=<tenant-id-canary> --apply --stop-on-error --json

4. Audit canary

Pastikan canary sudah up_to_date, lalu jalankan smoke test fitur terkait.

npm run tenant:schema:audit -- --tenant-ids=<tenant-id-canary> --json

C. Rollout semua tenant dengan batch

Naikkan batch secara konservatif. Mulai dari 10 tenant, observasi metrik, baru tambah ukuran batch bila migration ringan dan database stabil.

# Batch 1
npm run tenant:schema:deploy -- --limit=10 --offset=0 --apply --stop-on-error --json

# Audit batch 1
npm run tenant:schema:audit -- --limit=10 --offset=0 --json

# Batch 2
npm run tenant:schema:deploy -- --limit=10 --offset=10 --apply --stop-on-error --json

# Audit batch 2
npm run tenant:schema:audit -- --limit=10 --offset=10 --json

Artifact wajib

  • Output --json dari audit awal.
  • Output --json dari setiap batch deploy.
  • Output --json dari audit setelah batch.
  • Catatan tenant failed: tenantId, dbName, migration, error, waktu, dan keputusan fix.

Kapan batch boleh dinaikkan?

  • Tidak ada tenant failed.
  • Error rate aplikasi/worker normal.
  • DB lock dan connection pool stabil.
  • Smoke test fitur utama lulus.
  • Tidak ada alert dari WA, Instagram, realtime, atau worker yang terkait schema baru.

D. Jika migration tenant gagal

1. Stop dan isolasi

Karena memakai --stop-on-error, proses berhenti di tenant bermasalah. Jangan lanjut ke offset berikutnya sebelum penyebab jelas.

2. Diagnosis

Cek apakah error berasal dari lock, permission DB, migration history, data duplicate, constraint violation, long-running transaction, atau SQL destruktif.

3. Fix targeted

Lakukan fix paling kecil ke tenant tersebut. Hindari perubahan manual yang tidak terdokumentasi; jika perlu manual SQL, simpan sebagai incident note.

4. Retry tenant tunggal

npm run tenant:schema:deploy -- --tenant-ids=<tenant-id> --apply --stop-on-error --json
npm run tenant:schema:audit -- --tenant-ids=<tenant-id> --json

E. Post-deploy verification

  1. Jalankan audit final untuk seluruh tenant secara batch sampai semua tenant terverifikasi.
  2. Smoke test login REGULAR dan AGENCY.
  3. Smoke test instance WhatsApp, direct chat, blast canary, realtime update, dan worker queue jika schema terkait.
  4. Jika migration menyentuh Instagram, uji dashboard, OAuth status, media list, webhook verification, dan automation sample.
  5. Pastikan worker tidak menghasilkan spike failed/retry setelah schema berubah.
  6. Update release note dengan daftar migration, batch yang dijalankan, tenant failed/fixed, dan hasil audit final.

F. One-time legacy tools

Tidak dipakai untuk flow normal saat ini. tenant:schema:sync adalah alat catch-up berbasis db push untuk tenant lama yang drift. Karena tenant sudah up-to-date, release normal harus memakai audit → deploy → audit, bukan sync.

Audit baseline legacy

npm run tenant:migration-baseline:audit -- --limit=25 --json

Dipakai hanya jika ada tenant lama yang schema-nya benar tetapi history _prisma_migrations belum lengkap.

Baseline legacy

npm run tenant:migration-baseline:apply -- --tenant-ids=<tenant-id> --apply --stop-on-error --json

Hanya boleh setelah tenant tersebut terbukti up_to_date. Jangan baseline tenant yang masih drift.

Jangan mengandalkan rollback migration destruktif. Untuk partial rollout, aplikasi harus toleran terhadap beberapa versi schema. Pilih forward-fix untuk perubahan yang sudah menyentuh data; restore hanya dengan keputusan incident commander dan dampak tenant yang jelas.

Known operational limits

  • CLI sequential—aman namun rollout banyak tenant dapat lama; gunakan batch/offset dan observasi.
  • Registry menyimpan credential DB tenant dalam central DB; akses dan backup central harus sangat dibatasi.
  • Deteksi “destructive” berbasis pola SQL dan tidak menggantikan review DBA.
  • --allow-destructive adalah override tajam; wajib change approval, backup, dan rollback/forward-fix plan tertulis.
  • Provisioning baru memakai migration deploy dan advisory lock, tetapi cleanup orphan DB masih manual.
  • Selama rollout tenant berjalan, aplikasi dan worker harus siap menghadapi mixed schema version sementara.
14 · External integration

External Instance / Workspace API

Credential dibuat per instance, berprefix sl_live_inst_, secret diverifikasi dengan SHA-256 + timing-safe compare, dan scope tenant/instance/agency diambil dari credential—bukan dari request client.

Penyimpanan secret: API key disimpan terenkripsi AES-256-GCM agar UI dapat menampilkannya kembali, sekaligus secret hash untuk autentikasi. Lindungi EXTERNAL_API_KEY_ENCRYPTION_SECRET di secret manager dan rotasi dengan prosedur yang mempertimbangkan ciphertext lama.

Quick start

Authorization: Bearer sl_live_inst_<key-id>_<secret>
Content-Type: application/json
EndpointInput utamaPerilaku
GET /api/v1/external/instance/connectionBearerProbe status instance tanpa mengirim pesan.
GET /api/v1/external/audience/segmentsBearerList contact list milik instance.
POST /api/v1/external/audience/segments{ name }Membuat segment terikat instance.
GET /api/v1/external/audience/contactssegmentId,page,limit,searchPagination contact dan validasi segment milik instance.
POST /api/v1/external/audience/contactssegment + personal/group identityUpsert; soft-deleted contact dapat direstore.
POST /api/v1/external/messages/sendtargetType/value, message/attachmentDirect send setelah instance connected.

Contoh direct send

curl --request POST \
  --url https://<sevenlead-host>/api/v1/external/messages/send \
  --header 'Authorization: Bearer <API_KEY>' \
  --header 'Content-Type: application/json' \
  --data '{
    "targetType": "PERSONAL",
    "targetValue": "6281234567890",
    "message": "Halo dari sistem eksternal",
    "referenceId": "order-2026-0001"
  }'

Semantik saat ini

  • API key rotation merevoke key aktif sebelumnya.
  • lastUsedAt diperbarui pada autentikasi.
  • Contact create adalah idempotent/upsert.
  • referenceId hanya dikembalikan untuk trace; belum menjadi idempotency constraint.
  • Attachment base64 melewati request aplikasi dan WA service.

Consumer checklist

  • Simpan key hanya di backend/secret manager.
  • Gunakan timeout dan retry hanya untuk error transient.
  • Jangan blind retry direct send tanpa idempotency di sisi consumer.
  • Batasi ukuran attachment di sisi client.
  • Rotasi key saat bocor dan tangani 401 sebagai revoked/regenerated.
Production gap: route external belum terlihat memiliki rate limiting, quota per key, request idempotency, atau audit event per direct-send. Tambahkan sebelum exposure publik/high-volume.

Startup order

  1. PostgreSQL central dan tenant tersedia.
  2. Jalankan central migration, kemudian tenant rollout/audit.
  3. Start Sevenlead sehingga callback endpoint siap.
  4. Start realtime socket.
  5. Start WA service dengan persistent volume; bootstrap akan restore client.
  6. Start worker satu per satu dan observasi claim/error.
  7. Smoke test login, instance, direct chat, blast canary, realtime.

Membership/auth URLs, Socket public URL, Meta app/webhook secrets, external API encryption secret.
